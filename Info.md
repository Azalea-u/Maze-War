# Maze War — Rust Recreation

## Objective

Recreate Maze War, one of the first networked first-person shooters (1973), using modern Rust.
The goal is a faithful reimplementation: a first-person ray-cast view, a top-down minimap, and real-time multiplayer over UDP using a client-server architecture.
The project is structured across five phases, moving from a local rendering core outward to full networked play.

---

## Project Structure

Three crates in a single Cargo workspace:

- `common` — shared types: maze grid, player state, packet protocol
- `server` — UDP server, peer state management, game authority
- `client` — window, ray caster, minimap, HUD, network thread

---

## Phase 1 — Scaffold & Window

Get a working window with a pixel framebuffer and a functional CLI entry point.

**Tasks**

- Set up the Cargo workspace with `common`, `server`, and `client` crates
- Integrate `winit` + `pixels` for windowing and raw pixel rendering
- Implement the CLI entry sequence: prompt for server IP and username before opening the window
- Build an FPS counter using `std::time::Instant`, rendered top-left every frame

**Crates:** `winit`, `pixels`

---

## Phase 2 — Maze Generation & Minimap

Define the game world and get the player moving through it.

**Tasks**

- Define the maze as a flat `Vec<u8>` bitfield in `common`, where each byte encodes open passages as bits (N=1, S=2, E=4, W=8)
- Implement procedural maze generation using **Recursive Backtracker + Braiding** (see below)
- Define the `Player` struct: `pos: Vec2`, `angle: f32`, `alive: bool`, `score: u32`
- Implement WASD movement with grid collision
- Render a top-down minimap in a corner of the framebuffer showing the full maze, player position, and facing direction

**Generation Algorithm**

Mazes are generated by the server at startup using a two-step process:

1. **Recursive Backtracker (DFS)** — produces a perfect maze (exactly one path between any two cells) by carving passages with an iterative stack. Uses `Vec` as the stack; no recursion depth concerns.
2. **Braiding pass** — scans for dead ends (cells with only one open passage) and randomly removes a wall for each one with probability `braid_factor`. This punches loops into the perfect maze, reducing dead-end density.

Difficulty is controlled exclusively by `braid_factor`:

| Level | `braid_factor` | Character |
|-------|---------------|-----------|
| 1 | 0.65 | Few dead ends, open corridors, easy navigation |
| 2 | 0.25 | Moderate dead ends, balanced layout |
| 3 | 0.00 | No braiding — perfect maze, maximum dead ends |

The generated maze is sent to clients inside `ServerSnapshot` as a flat serialized `Vec<u8>` via `bincode`. Each level seeds the RNG freshly, so every game session gets a unique maze.

**Crates:** `rand`

## Phase 3 — Ray Caster (3D View)

Implement the core first-person renderer using DDA ray casting.

**Tasks**

- Cast one ray per screen column using the DDA algorithm
- Compute perpendicular wall distance and derive column draw height: `screen_h / perp_dist`
- Shade walls by distance (farther = darker), black floor and ceiling
- Store a per-column Z-buffer for sprite depth testing
- Render other players as a scaled eyeball sprite, clipped correctly against the Z-buffer
- Verify the renderer sustains above 50 fps consistently

> Reference: [Lode's Raycasting Tutorial](https://lodev.org/cgtutor/raycasting.html).
> The DDA loop is roughly 30 lines of Rust. Sprite Z-sorting reuses the same distance buffer from the ray pass.

## Phase 4 — Server & UDP Networking

Implement the client-server architecture over UDP.

**Packet Protocol** (defined in `common`)

```rust
enum Packet {
    Join    { name: String },
    State   { pos: Vec2, angle: f32, alive: bool },
    Shoot,
    ServerSnapshot { peers: Vec<PeerState>, maze: Vec<u8>, level: u8 },
}
```

Serialized with `bincode`.

**Server**

- Tokio UDP server accepting a minimum of 10 simultaneous clients
- Maintains a `HashMap<SocketAddr, PeerState>`
- Broadcasts a `ServerSnapshot` to all clients on each tick
- Validates `Shoot` packets: checks ray vs all peer positions, updates scores, sends hit confirmation
- Selects and sends the active level to clients on join

**Client**

- Spawns a dedicated tokio task for UDP recv/send
- Shares peer state with the render loop via `Arc<Mutex<PeerMap>>`
- Sends local `State` packet at ~20–30 Hz
- Renders at 60 Hz using latest known peer positions (no interpolation required)

**Crates:** `tokio`, `serde`, `bincode`, `rand`

---

## Phase 5 — Polish & Integration

Complete the HUD, level selection, respawn logic, and multi-machine testing.

**Tasks**

- HUD: FPS top-left, score top-right, username above visible peers
- Level selection: server receives level (1–3) as a startup argument, generates the maze accordingly, and includes both the level ID and serialized maze in `ServerSnapshot`
- Respawn: on death, wait 3 seconds, spawn at a random open cell managed by the server
- Full multi-machine test: server on one machine, two or more clients connecting from different IPs; verify state sync, hit detection, and frame rate under load

---

## Performance Target

The renderer must sustain above 50 fps at all times.
A software ray caster writing directly into a `pixels` framebuffer is CPU-bound and well within this target on any modern machine.
Profile if needed; the ray loop and sprite pass are the only hot paths.
